## 큰 흐름의 상태코드 - 1xx (Informational) : 요청이 수신되어 처리중 (잘 사용하지 않음) - 2xx (Successful) : 요청 정상 처리 - 3xx (Redirection) : 요청을 완료하려면 추가 행동이 필요 - 4xx (Client Error) : 클라이언트 오류, 잘못된 문법등으로 서버가 요청을 수행할 수 없음 - 5xx (Server Error) : 서버 오류, 서버가 정상 요청을 처리하지 못함 --- ## 많이 사용되는 상태코드 ### 2xx (Successful) - 200 : Ok , 일반적인 요청 성공 - 201 : Created , 요청 성공해서 새로운 리소스가 생성 - 컬렉션 형식으로 리소스 생성 후 헤더 필드에 Location 필드를 추가 후 식별 - ex) Location : /members/100 -> 100 은 서버가 생성 한 리소스 URI - 202 : Accepted , 요청이 접수되었으나 처리가 완료되지 않았음 - 배치 처리 같은 곳에서 사용 - 204 : No Content , 서버가 요청을 성공적으로 수행했지만, 응답 페이로드 본문에 보낼 데이타가 없음 - ex) 웹 문서 편집기에서 SAVE 버튼 - save 버튼의 결과로 아무 내용이 없어도 된다. - save 버튼을 눌러도 같은 화면을 유지해야 한다. - 결과 내용이 없어도 204 메시지 만으로 성공을 인식할 수 있다. - 더 쉽게 예를 들면, 웹 페이지에서 내용을 수정하고 저장 버튼을 눌렀을 때 클라이언트 측에서는 저장이 되었다. 라는 것만 알면 될 뿐 서버에서 따로 메시지 바디로 결과 값을 내려 줄 필요가 없을 때 사용 ## 3xx (Redirection) - 300 Multiple Choices , 거의 사용 X - 301 Moved Permanently , 리다이렉트 요청 시 메서드가 GET으로 변하고, 본문이 제거될 수 있음 (MAY) - ![[301.jpeg]] - 302 Found , 리다이렉트 요청 시 메서드가 GET으로 변하고, 본문이 제거될 수 있음 (MAY) - 303 See Other , 302와 기능은 같지만, 리다이렉트 요청 시 메서드가 GET으로 변경 - 304 Not Modified , 캐시를 목적으로 사용, 클라이언트에게 리소스가 수정되지 않았음을 알려준다. 따라서 클라이언트는 로컬 PC에 저장된 캐시를 재사용한다. (캐시로 리다이렉트 함) - 304 응답은 응답에 메시지 바디를 포함하면 안 된다. (로컬 캐시를 사용하므로) - 조건부 GET, HEAD 요청 시 사용 - 307 Temporary Redirect , 302와 기능은 같으나, 리다이렉트 요청 시 메서드와 본문 유지 (요청 메서드를 변경하면 안된다. MUST NOT) - 308 Permanent Redirect , 301과 기능은 같지만, 리다이렉트 요청 시 메서드와 본문 유지 (처음 POST를 보내면 리다이렉트도 POST 유지) , 웬만하면 301 사용 - ![[308.jpeg]] #### 리다이렉션 이해 - 웹 브라우저는 3xx 응답의 결과에 Location 헤더가 있으면, Location 위치로 자동 이동(리다이렉트) - 자동 리다이렉트는 서버에서 옛날에 사용하던 event 라는 페이지에 클라이언트가 접속을 했을 때 자동으로 new-event 라는 페이지로 리다이렉트 시켜주는 방식을 뜻하며, 이는 new-event 라는 페이지로 들어갈 때 서버에 API를 한 번 더 자동으로 요청을 한 뒤 200 ok 신호를 받음 #### 종류 - 영구 리다이렉션 - 특정 리소스의 URI가 영구적으로 이동 - ex) /members -> /users - ex) /event -> /new-event - 일시 리다이렉션 - 일시적인 변경 - 주문 완료 후 주문 내역 화면으로 이동 - PRG : Post/Redirect/Get - ex) POST로 주문 후 웹 브라우저를 새로고침 시 POST를 다시 요청하여 중복 주문이 될 수 있는 문제점 - 이런 문제점을 해결하기 위해 서버에서 중복 된 주문 번호가 들어올 시 처리를 하는 방법도 있지만, 클라이언트 쪽에서 POST로 주문을 한 뒤 새로고침으로 인한 중복 주문 방지 방법은 POST로 주문후에 주문 결과 화면을 GET 메서드로 리다이렉트 결론적으로 새로고침을 해도 결과 화면을 GET으로 조회, 중복 주문 대신에 결과 하면만 GET으로 다시 요청 - 정리를 하면 결과 화면에서 새로고침 시 결과 화면으로 GET 요청 처리 - ![[PRG.jpeg]] - 특수 리다이렉션 - 결과 대신 캐시를 사용 ## 4xx (Client Error) - 400 Bad Request , 클라이언트가 잘못된 요청을 해서 서버가 요청을 처리할 수 없음 - 요청 구문, 메시지 등 오류, 클라이언트는 요청 내용을 다시 검토하고, 보내야 함 - ex) 요청 파라미터가 잘못되거나, API 스펙이 맞지 않을 때 - 401 Unauthorized , 클라이언트가 해당 리소스에 대한 인증이 필요함 - 인증(Authentication) 되지 않음을 뜻함 - 401 오류 발생 시 응답에 WWW-Authenticate 헤더와 함께 인증 방법을 설명 - 참고 - 인증(Authentication) : 본인이 누구인지 확인 (로그인) - 인가(Authorization) : 권한부여 (ADMIN 권한 처럼 특정 리소스에 접근 할 수 있는 권한, 인증이 있어야 인가가 있음) - 403 Forbidden , 서버가 요청을 이해했지만, 승인을 거부함 - 주로 인증 자격 증명은 있지만, 접근 권한이 불충분한 경우 - ex) ADMIN 등급이 아닌 사용자가 로그인은 했지만, ADMIN 등급의 리소스에 접근하는 경우 - 404 Not Found , 요청 리소스를 찾을 수 없음 - 요청 리소스가 서버에 없음 - 또는 클라이언트가 권한이 부족한 리소스에 접근할 때 해당 리소스를 숨기고 싶을 때 (403을 사용해서 처리해도 되지만, 이 사이트를 숨기고 싶을 때 사용) ### 5xx (Server Error) - 500 Internal Server Error , 서버 문제로 오류 발생, 애매하면 500 오류 - 503 Service Unavailable , 서비스 이용 불가 - 서버가 일시적인 과부화 또는 예정된 작업으로 잠시 요청을 처리할 수 없음 - Retry-After 헤더 필드로 얼마뒤에 복구되는지 보낼 수도 있음 - 웬만하면 5xx 번 대 에러는 만들면 안 된다. 진짜 중요할 때만 만드는걸 권장한다.
