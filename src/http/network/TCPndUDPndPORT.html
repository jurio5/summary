- IP의 문제점을 해결하기 위해 이 TCP 라는 신뢰성을 보장하는 프로토콜이 만들어졌다. - 정확히는 TCP는 IP 위에서 동작하는 별개의 전송 계층 프로토콜이다. - 이 TCP는 `3way handshake`을 통해 IP의 문제점들을 모두 해결할 수 있다. - 기본적으로 IP를 사용할 때 가장 문제가 되었던 `비신뢰성`,`비연결성`,`순서보장`등의 문제를 이 `3way handshake`를 통해 해결 할 수 있는데, 그 방법은 클라이언트 쪽에서 `synchronized`의 약자인 `syn`을 연결을 전달하고자 하는 서버 쪽으로 보내게 되는데 서버 쪽에서 문제가 생기지 않았다. 라는 것을 요청 한 클라이언트에게 알려주기 위해 `syn` 신호와 함께 `Acknowledgement`의 약자인 `ACK`신호를 같이 돌려준다. 이렇게 되면 일단, `비연결성`의 문제는 해결되면서 `비신뢰성`의 문제까지 함께 해결이 되는 것이다. - 나머지는 `순서보장`의 문제인데 이 순서보장의 문제를 어떻게 해결하냐. 라는 부분에서는 클라이언트가 `안녕`,`안녕1`,`안녕2` 라는 메시지를 서버에게 보낼 때 헤더 쪽에 시퀀스 번호를 포함해서 여러가지 정보를 담아서 서버 쪽으로 데이터를 보내면 서버 쪽에서 그 헤더에 대한 정보를 보고 `안녕`,`안녕2`라는 순서로 패킷이 잘못 들어왔을 경우 다시 클라이언트에게 재전송을 요구하게 된다. 이 재전송 방식은 여러가지 방식이 있는데 Stop & wait , Sliding Window, Go Back N ARQ 등 흐름제어, 오류제어, 혼잡제어 같은 방식을 통해 재전송을 요청하게 되면서 이런 방식을 통해 `순서보장`의 문제가 해결이 된다. - 그렇다면 여기서 문제가 되는 부분은 동일한 아이피로 여러가지 데이터를 서버에 보내게 된다면 서버 쪽에서는 어떤 방식으로 클라이언트에게 수신 된 알맞는 데이터에 맞게 송신할 수 있을까? 라는 부분에서 `PORT`라는 개념이 존재한다. - 클라이언트에서 `안녕` 이라는 데이터를 보낸다고 칠 때 `OS`계층에서 TCP를 한 번 거치게 되는데 이 때 `PORT`라는 것을 함께 담아서 서버 쪽으로 보내게 된다. 이 때 TCP에서 출발지 포트와 목적지 포트를 함께 담아서 IP 쪽으로 넘기면서 IP에서는 그 TCP 정보 위에 출발지IP 및 목적지IP 그리고 여러가지 정보들을 담아서 네트워크 인터페이스 계층으로 넘기게 되고, 여기서 이더넷 프레임을 통해 MAC 주소의 정보를 한 번 더 감싼 뒤 서버 쪽으로 보내고, 서버는 이 정보들을 열어보며, 이 전송 된 데이터를 다시 클라이언트에게 알맞는 포트 주소로 넘겨주게 되면서 클라이언트 사용자의 화면에 렌더링이 되는 구조이다. - UDP의 경우 현재 HTTP 3 버전에서 각광 받으며 뜨고 있는 기술이지만, 현재로서는 IP와 동일하게 비신뢰성 비연결성 같은 특징들을 가지고 있고, 매우 빠른 기술이다. 정도로만 알면 될 것 같고, 곁가지 더 붙여보자면 UDP는 별도의 흐름 제어나 오류 제어가 없어서 애플리케이션에서 추가적인 처리를 해야하며, 새하얀 도화지와 같은 상태라서 요즘에는 여기에 정보들을 데코레이션 하면서 사용을 하기도 한다.
