### 데이터베이스 커넥션 획득 과정 - 데이터베이스 커넥션을 획득할 때는 다음과 같은 복잡한 과정을 거침 1. 애플리케이션 로직은 DB 드라이버를 통해 커넥션을 조회 2. DB 드라이버는 DB와 `TCP/IP`커넥션을 연결, 물론 이 과정에서 `3 way handshake`같은 `TCP/IP`연결을 위한 네트워크 동작이 발생 3. DB 드라이버는 `TCP/IP`커넥션이 연결되면, ID, PW와 기타 부가정보를 DB에 전달 4. DB는 ID, PW를 통해 내부 인증을 완료하고, 내부에 DB 세션을 생성 5. DB 커넥션 생성이 완료되었다는 응답을 보냄 6. DB 드라이버는 커넥션 객체를 생성해서 클라이언트에 반환 ![[커네션 풀 연결 과정.jpg]] - 이런 과정에서 DB로 커넥션을 할 때 `TCP/IP`를 통해 연결을 시키고 DB 내부에서 인증을 하고 반환하는 과정을 거치면서 커넥션을 새로 만들기에 리소스를 매 번 사용해야한다. - 클라이언트 쪽에서 요청을 할 때 SQL을 실행하는 시간 뿐만 아니라 커넥션 풀을 만들고 반환하고 하는 일련의 과정들이 전부 다 사용자 경험으로 고스란히 전달되면서 불편함을 느끼게 된다. - 이러한 문제를 한 번에 해결하는 방법은 스레드 풀을 생성해두는 방법처럼 커넥션을 미리 생성해두고 사용하는 `커넥션 풀`방식을 사용한다. ### 커넥션 풀 정리 - 적절한 커넥션 풀 숫자는 서비스의 특징과 애플리케이션 서버 스펙 및 DB 서버 스펙에 따라 다르기 때문에 성능 테스트를 통해서 정해야 함 - 커넥션 풀은 서버 당 최대 커넥션 수를 제한할 수 있음. 따라서 DB에 무한정 연결이 생성되는 것을 막아주기에 DB를 보호하는 효과도 있음 - 이런 커넥션 풀은 얻은 이점이 매우 크기 때문에 `실무에서는 항상 기본으로 사용` - 기본으로 `10`개 정도를 사용하지만, 서버 스펙에 따라서 정해야 함 - 커넥션 풀은 개념적으로 단순해서 직접 구현할 수도 있지만, 사용도 편리하고 성능도 뛰어난 오픈소스 커넥션 풀이 많기 때문에 오픈소스를 사용하는걸 지향 - 대표적인 커넥션 풀 오픈소는 `commons-dbcp2`, `tomcat-jdbc pool`, `HikariCP`등이 있음 - 성능과 사용의 편리함 측면에서 최근 트랜드는 `hikariCP`를 주로 사용하며, 스프링 부트 2.0 부터는 기본 커넥션 풀로 `hikariCP`를 제공 - 결론은 레거시 프로젝트가 아니라면 커넥션 풀로는 `hikariCP`를 사용하자
