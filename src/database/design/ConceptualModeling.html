## 개념적 모델링 ### 개념적 모델링이란 - 데이터 세상의 청사진을 그리는 단계로, 현실 세계의 비즈니스를 데이터 구조로 표현하는 과정이다. - 기술적인 용어보다 **모두가 이해할 수 있는 그림과 용어로 합의**하며, 비즈니스의 본질을 데이터로 구체화한다. - 개발자, 기획자, 현업 담당자가 함께 참여한다. #### 명사와 동사 분석 - **명사 → 엔티티(Entity)** 또는 **속성(Attribute)** - 예 : 회원,상품,주문 등 - **동사 → 관계(Relationship)** - 동사 예 : 주문하다, 결제하다 등 - 관계 예: 회원은 상품을 주문한다 → 회원–주문–상품 관계 ### 엔티티(Entity) #### 정의 - 데이터를 저장하고 관리해야 할 독립적인 대상. - 엑셀로 비유하면 하나의 시트(sheet)에 해당한다. #### 좋은 엔티티의 조건 1. 업무적으로 관리할 가치가 있어야 한다. 2. 고유하게 식별 가능해야 한다. 3. 두 개 이상의 속성을 가진다. 4. 인스턴스(행)의 집합이다. 5. 다른 엔티티와 관계를 맺는다. #### 엔티티 예시 | 엔티티 | 속성 | 식별자 | | --- | ------------------------- | ----- | | 회원 | 아이디, 이름, 주소, 연락처, 가입일 | 아이디 | | 상품 | 상품 코드, 이름, 가격, 재고 | 상품 코드 | | 주문 | 주문번호, 주문일시, 상태, 배송지, 결제금액 | 주문번호 | --- ### 엔티티 분류 #### 존재 형태 기준 | 구분 | 특징 | 예시 | | ------ | -------------- | ---------- | | 유형 엔티티 | 물리적 실체가 있는 대상 | 회원, 상품, 건물 | | 개념 엔티티 | 추상적인 개념 | 계좌, 학과 | | 사건 엔티티 | 시간에 따라 발생하는 사건 | 주문, 예약, 결제 | #### 역할 및 발생 시점 기준 |구분|설명|예시| |---|---|---| |기본 엔티티|독립적으로 존재|회원, 상품| |중심 엔티티|기본 엔티티로부터 파생|주문| |행위 엔티티|중심 엔티티의 상세 기록|주문 이력, 결제 내역| #### 독립성 기준 |구분|설명|예시| |---|---|---| |강한 엔티티|독립적으로 존재하며 고유한 PK를 가짐|회원, 상품| |약한 엔티티|부모 엔티티에 종속되어 존재|부양가족(→사원)| ### 속성과 식별자 **속성(Attribute)** - 엔티티가 가지는 구체적 정보 (테이블의 컬럼에 해당) **식별자(Identifier)** - 인스턴스를 고유하게 구분하는 값 (Primary Key) | 엔티티 | 주요 속성 | 식별자 | | --- | --------------------------- | ----- | | 회원 | 아이디, 비밀번호, 이름, 주소, 연락처, 가입일 | 아이디 | | 상품 | 상품 코드, 이름, 가격, 재고, 등록일 | 상품 코드 | | 주문 | 주문번호, 주문일시, 상태, 배송지, 결제금액 | 주문번호 | ### 관계(Relationship) #### 관계의 종류 (카디널리티) |유형|설명|예시| |---|---|---| |1:1|서로 하나씩 대응|주문–배송| |1:N|한쪽이 여러 개|회원–주문| |N:1|반대 방향|주문–회원| |M:N|양쪽 모두 여러 개|주문–상품| #### 참여도(Optionality) - **필수(Mandatory)**: 반드시 관계를 가져야 함 (예: 주문 → 회원) - **선택(Optional)**: 없어도 됨 (예: 회원 → 주문) ### ERD(Entity-Relationship Diagram) #### 역할 - 엔티티, 속성, 관계, 카디널리티, 참여도를 시각적으로 표현하는 도구. 복잡한 구조를 한눈에 파악할 수 있게 해준다. #### 까마귀발 표기법 (실무 표준) | 기호 | 의미 | | --- | ------- | | ` | ` | | `<` | N | | `O` | 0 (선택적) | 예시 관계: - `회원 (||) — (O<) 주문` - `주문 (>O) — (|<) 상품` ### 핵심 요약 - 개념적 모델링은 데이터베이스 설계의 첫 단계 - 현실의 비즈니스를 데이터 구조로 시각화 - 엔티티 → 속성 → 관계 순으로 설계 - ERD를 통해 설계 청사진을 공유하고 합의
