## 설계의 중요성 ### 데이터베이스 설계 - 설계는 **집을 짓기 전의 설계도와 같다.** - 설계 없이 만들면 초반엔 빠를 수 있지만, 시간이 지날수록 **기술 부채(Technical Debt)**가 쌓인다. - 체계적 설계는 **데이터 중복 제거, 무결성 유지, 성능 최적화, 유지보수 용이성**을 확보하는 핵심 과정. > “좋은 설계는 비용이 아니라 투자다.” --- ### 잘못된 설계가 부르는 재앙 #### 문제 상황 - 하나의 테이블(`orders`)에 고객, 상품, 주문 정보를 모두 저장. ``` `order_id | customer_id | customer_name | customer_address | product_id | product_name | product_price | ordered_at` ``` #### 3대 문제점 1. **데이터 무결성 훼손** - 데이터 중복으로 인해 이상 현상(Anomaly) 발생 - **수정 이상 (Update Anomaly)** → 일부 데이터만 수정 시 불일치 발생 - **삽입 이상 (Insertion Anomaly)** → 불필요한 정보 없이는 원하는 데이터 삽입 불가 - **삭제 이상 (Deletion Anomaly)** → 한 정보를 지우면 관련된 다른 정보까지 사라짐 2. **성능 저하** - 테이블이 비대해져 조회 시 **Full Scan** 증가 → 속도 저하 3. **유지보수 비용 증가** - 구조 변경이 어렵고, 작은 수정도 대규모 영향 발생 - DB의 문제를 코드로 해결 → **애플리케이션 복잡도 상승** --- ### 설계의 3단계 #### 개념적 설계 (Conceptual Design) - **목표:** 비즈니스 요구사항 파악, 핵심 데이터(Entity) 및 관계(Relationship) 정의 - **산출물:** ERD(Entity-Relationship Diagram) - **비유:** 건축가의 스케치 #### 논리적 설계 (Logical Design) - **목표:** 관계형 DB 이론에 맞게 구조화, 정규화 수행 - **산출물:** 정규화된 테이블 스키마 (PK, FK 정의) - **비유:** 건축 설계도 (청사진) #### 물리적 설계 (Physical Design) - **목표:** 특정 DBMS(MySQL 등)에 최적화하여 구현 - **산출물:** 실제 `CREATE TABLE` SQL, 인덱스, 데이터 타입 지정 - **비유:** 시공 계획서 #### 각 모델의 관점 - **개념 모델:** 현실 세계 관점 — 비전문가와 소통 중심 - **논리 모델:** 관계형 DB 이론 관점 — 정규화 중심 - **물리 모델:** 실제 구현 관점 — 데이터 타입, 인덱스 등 최적화 - 실무에서는 용어를 엄격히 구분하지 않고 **혼용**하는 경우도 많음.
